## 0. 一句话商业模式

> **用「便宜到约等于免费」的 DeepSeek Flash 做获客与留存（每天 ¥3 免费额度分 5h 批次发、几乎用不完），用「贵但强」的 Claude / GPT / Gemini 做付费墙；高端模型必须充钱开通余额计费 session；会员预付制把用户从「按次心疼」变成「包月安心」，让利 25% 换黏性与现金流。**

这套模式的核心洞察：**Flash 太便宜了**。¥3/天在 Flash 上够发几百轮对话——对一个个人学习管家来说等于无限。所以免费额度不是"限制"，而是"获客诱饵"：用户感觉永远免费、天天来。

真正的成本（也就是付费墙）发生在用户**主动切到高端模型**时——Claude Opus / GPT / Gemini 单价是 Flash 的 50~100 倍，一轮深度对话就可能花掉 ¥1~5。这时 ¥3/天瞬间见底 → 自然弹出「开会员 or 按量充值」。

```
                  ┌─────────────────────────────────────────┐
  获客/留存  ──▶  │ Flash 免费层（¥3/天，几乎用不完）         │  ← 病毒系数、日活
                  └─────────────────────────────────────────┘
                                   │ 用户想要更聪明的答案
                                   ▼
                  ┌─────────────────────────────────────────┐
  变现       ──▶  │ 高端模型（Claude/GPT/Gemini）= 付费墙      │  ← ARPU
                  │   · 按量钱包（成本×1.2）                   │
                  │   · 会员包月（折合便宜 ~30% + 全模型解锁） │
                  └─────────────────────────────────────────┘
```

## 1. 成本基线（决定一切定价）

> ⚠️ 下表为**代表性单价**（USD / 百万 token，约 ¥7.2/USD）。真实值随供应商调价波动——
> 本方案配套的**多供应商网关**会顺带把各家 `pricing` 拉成可配置表（见 §4），定价不写死。

| 模型 | 输入 $/M | 输出 $/M | 折 ¥/千token(输出) | 相对 Flash |
|---|---|---|---|---|
| **DeepSeek V4 Flash**（默认/免费层） | ~0.14 | ~0.28 | ~¥0.002 | 1× |
| DeepSeek V4 思考 | ~0.28 | ~0.42 | ~¥0.003 | ~1.5× |
| **Gemini 2.5 Pro** | ~1.25 | ~10 | ~¥0.072 | ~36× |
| **GPT（4o/5 级）** | ~2.5 | ~10 | ~¥0.072 | ~36× |
| **Claude Sonnet** | ~3 | ~15 | ~¥0.108 | ~54× |
| **Claude Opus** | ~15 | ~75 | ~¥0.54 | ~270× |

**关键结论**：
- ¥3/天 ÷ Flash(¥0.002/千输出) ≈ **每天 150 万输出 token** → 普通用户**绝无可能**在 Flash 上花完 → 免费层=零成本获客。
- 同样 ¥3 在 Claude Opus 上 ≈ **5,500 token** ≈ **1~2 轮**深度长对话 → 高端模型天然是付费墙。

## 2. 免费层 spec：每天 ¥3 真实成本额度

### 2.1 计量层（前端，localStorage，5h 滑动窗口）

新建 `lib/usage.ts`：

```ts
// 每个模型的「成本单价」表（¥/千token），可被 §4 网关拉取的实时 pricing 覆盖
COST_PER_K: Record<AiModelId, { in: number; out: number }>

// chat 路由在流尾返回 usage（prompt_tokens / completion_tokens）→
// 客户端 recordUsage(modelId, promptTokens, completionTokens)
//   cost = promptTokens/1000*in + completionTokens/1000*out
//   累加到 localStorage["butler.usage.<windowStart>"]

DAILY_FREE_BUDGET = 3.0            // ¥/天
WINDOW_HOURS = 5                   // 5h 一批
WINDOW_BUDGET = 0.6               // ¥/窗口（≈ 3 ÷ 4.8 窗/天）
getWindowSpend(): number          // 当前窗口已花 ¥
getWindowRemaining(): number      // max(0, 0.6 - 当前窗口已花)
getNextResetAt(): number          // 当前窗口结束时间戳（倒计时用）
canSpend(estCost): boolean        // 发送前预检（看当前窗口余量）
```

### 2.1b 5 小时滑动窗口（核心：分批发放）

- **不再一次给满 ¥3**：按 5 小时滑动窗口发放，每窗 ~¥0.6，**用不完不累积**（窗口翻篇即清零，新窗满血）。
- 窗口起点 = `floor(now / 5h) * 5h`（UTC 对齐，跨午夜/时区无缝）。一天约 4.8 个窗口，日封顶仍 ≈ ¥3。
- **作用**：① 防一口气烧光当天额度后整天无免费可用 → 流失；② 制造「再等 X 小时回满血」的**回访节律**（巴甫洛夫，呼应 [065] 每日仪式）；③ 高端模型烧得快→窗口几条就空→更快够到付费墙。
- 窗口用尽 ≠ 当天用尽：仍可切回 Flash（Flash 也计窗口，但 ¥0.6 在 Flash 上是几百轮，等于不限），或充钱包 / 开会员。

### 2.2 chat 路由改造（关键前置）

当前 `app/api/chat/route.ts` 是流式、**没回传 usage**。需要：
1. 在 SSE 流尾追加一条 `{ type: "usage", prompt_tokens, completion_tokens, model }` 事件（OpenAI/DeepSeek 流在 `stream_options: { include_usage: true }` 下最后一个 chunk 带 usage）。
2. 客户端 onReceive usage → `recordUsage(...)`。
3. 非流式路由（extract-ddls / generate-panel / research 等）同样累计——它们也烧 token，得计入额度。

### 2.3 额度耗尽后的体验（softwall，不粗暴）

```
本时段免费额度（¥0.6 / 5h 窗口）已用完。
  ┌──────────────────────────────────────┐
  │  你这阵和 Butler 聊得火热 🔥           │
  │  本时段额度已用完，2小时48分后回满血。 │  ← getNextResetAt 倒计时
  │  现在想继续：                          │
  │   ① 切回 DeepSeek Flash（依然免费）★   │  ← 默认引导，零成本留人
  │   ② 充值钱包，按量继续（成本×1.3）     │
  │   ③ 开会员，更高窗口额度 + 全模型解锁   │  ← 主推
  └──────────────────────────────────────┘
```

> 设计要点：**永远给一条"继续免费"的出口**（切回 Flash）。绝不让用户撞墙离开——撞墙=流失。付费是"想要更好"的主动选择，不是"被锁住"的被迫。

### 2.4 顶栏额度可视化

TopBar 加一个**极简额度条**（仅在本窗口已花 > 0 时出现）：`本时段 ¥0.4 / ¥0.6 · 1h12m 后回满` 细进度条 + 倒计时。让用户对消耗有感知 → 既建立"额度是真东西"的认知，又在窗口接近用完时自然引导升级；倒计时则把"回满血"变成回访钩子。

## 3. 付费层 spec：按量钱包 + 会员让利

### 3.1 按量钱包（pay-as-you-go）

- 用户预充值进「钱包余额」（¥10/¥50/¥100 档）。
- 每次请求按 **真实成本 × 1.3**（30% 加价 = 毛利）从钱包扣。覆盖网关/带宽/支付通道 + 利润。
- 透明：每条消息可展开看「本条花了 ¥0.09（Claude Sonnet · 2.1k token）」。
- 钱包优先级：**当前窗口免费额度 → 钱包余额 → 提示充值/开会员**。

### 3.2 会员（在现有 Free/Pro/Max 上重构）

现有 `lib/billing.ts` 是 Free ¥0 / Pro ¥39 / Max ¥99（月），年付 8 折。重构为「**包月额度 + 让利**」模型：

| | Free | **Pro ¥39/月** | **Max ¥99/月** |
|---|---|---|---|
| 每日免费额度（Flash） | ¥3 | ¥3 | ¥3 |
| 高端模型 | ❌（仅 Flash） | ✅ 全解锁 | ✅ 全解锁 |
| **含高端模型额度** | — | 折合零售价值 **~¥52**（39÷0.75） | 折合 **~¥132**（99÷0.75） |
| 折合单价 vs 按量 | — | **便宜 ~25%** | **便宜 ~25%** |
| 超额后 | 切 Flash / 充钱包 | 自动转按量(×1.3) | 自动转按量(×1.3) |
| 其他权益 | 基础 | 优先响应/历史不限 | + 并行调研/面板应用无限/抢先体验 |

**让利逻辑落地**：Pro ¥39 包含的高端模型用量，其**零售当量** = 39 / (1−0.25) = **¥52**。即用户花 ¥39 拿到平时按量要花 ¥52 的额度 → 字面"便宜 25%"，且**预付锁定现金流 + 沉没成本拉黏性**（已付月费 → 更舍得用 → 更离不开）。

> 数字口径（v1.1 已定）：按量 **×1.3**（毛利 30%）、会员让利 **25%**。比初版（×1.2 / 让利 30%）整体上调利润、收窄让利，仍保「会员比按量省」的黏性。后续按转化/留存数据再微调。

### 3.3 三条变现路径的分工

| 路径 | 服务人群 | 作用 |
|---|---|---|
| 免费 Flash | 所有人 | 获客 + 日活 + 口碑 |
| 按量钱包 | 轻度/尝鲜高端模型 | 低门槛转化、无承诺 |
| 会员包月 | 重度/认同产品 | **主利润 + 现金流 + 黏性** |

## 4. 自动化多供应商网关 spec

> 目标：把 `ai-models.ts` 里 Claude/GPT/Gemini 三个 `placeholder` 变成**真实可调**；chat 路由按所选 model 路由到对应供应商；并把各家**模型清单 + 定价**做成可自动拉取。
> **v1.1 关键约束**：三家高端 API **不默认开放**——用户必须先**充钱 → 在后台开通 → 系统为他开一个余额计费 session** 才能调用（机制见 §4.0）。

### 4.0 充钱 → 后台开通 → 余额计费 session（高端模型准入）

这是高端模型的「准入闸」，也是按量计费的执行体。流程：

```
用户想用 Claude/GPT/Gemini
   │
   ▼  ① 后台「高端模型」页（用户自己去开，默认关）
点「开通」→ 提示需充值
   │
   ▼  ② 充钱（¥10/¥50/¥100 进钱包余额）
余额 > 0 → 自动化流程触发
   │
   ▼  ③ 服务端为该用户开一个「计费 session」
   server 持平台统一 key（ANTHROPIC/OPENAI/GEMINI_API_KEY，绝不下发）
   session 记录：userId · 余额快照 · 已用 · 状态(active/exhausted)
   │
   ▼  ④ 用户每次调高端模型 → 走 session：
   平台 key 调供应商 → 拿 usage → 成本×1.3 从 session 余额扣
   余额归零 → session 置 exhausted → 自动降级 Flash + 提示再充值
```

要点：
- **用户永远不接触真实 key**：平台用自己的统一 key 调三家，用户只看到「余额」和「本条花了多少」。平台赚 ×1.3 的差价。
- **「后台再去开」= 显式准入**：高端模型默认锁，用户在后台主动开通（避免误触烧钱、也是一道转化确认）。会员（Pro/Max）则自动具备开通资格、用包月额度替代充值。
- **session 是计费边界**：一次充值 = 一段可用余额；余额是**硬上限**，绝不垫付（§6 风险）。演示阶段 session 存 localStorage，接真后端时迁服务端权威记账（防前端篡改）。
- **自动化流程** = ③④ 这套「充值到账 → 开 session → 路由计费 → 耗尽降级」全自动，无需人工。

### 4.1 Provider 抽象层

新建 `lib/providers/`：

```
lib/providers/
  types.ts          // Provider 接口：chat(stream) / listModels() / pricing()
  deepseek.ts       // 现有，OpenAI SDK baseURL=deepseek
  anthropic.ts      // Claude，@anthropic-ai/sdk
  openai.ts         // GPT，openai SDK 官方 baseURL
  google.ts         // Gemini，@google/generative-ai
  registry.ts       // modelId → provider 路由表 + env key 映射
```

- 各家 env key：`DEEPSEEK_API_KEY`（已有）/ `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` / `GEMINI_API_KEY`。
- 缺 key 的供应商：模型在下拉里灰显「未配置」（而非现在的"敬请期待"），配了 key 自动点亮。
- chat 路由：`const provider = registry.resolve(modelId)` → 统一 stream 接口 → 各家差异（Claude 的 system 单列、Gemini 的 contents 格式、tool schema 差异）在 provider 内部消化，路由层不感知。

### 4.2 「自动化获取」= 模型清单 + 定价自动发现

新建脚本 `scripts/sync-models.ts`（可手动跑 / CI 定时跑）：

1. 调各家 `/models` 接口（Anthropic `GET /v1/models`、OpenAI `GET /v1/models`、Google `models.list`）拉**最新可用模型**。
2. 拉**官方定价**（部分需维护一张 pricing 映射，因为 /models 不一定返回价格）。
3. 生成/更新 `lib/ai-models.generated.ts`（注册表 + COST_PER_K），供 §2 计量层与下拉直接消费。
4. 好处：供应商出新模型 / 调价 → 跑一次脚本就同步，不用手改代码。

> 这一步同时**喂养 §1 成本基线**——定价不写死、随官方变，计量永远准。

### 4.3 安全

- 所有 key 只在**服务端**（route / 脚本）读 `process.env`，绝不下发前端。
- 沿用现有 connector 路由的 SSRF 守卫思路。
- 计费在服务端做权威记账（演示阶段先前端 localStorage，接真后端时迁服务端）。

## 5. 增长杠杆（让"赚钱模式"转得动）

光有付费墙不够，得有流量灌进漏斗：

1. **免费层够慷慨 + 回访节律** → 口碑/自传播（[045] 分享卡 / PWA）。Flash 上每 5h 窗口 ~¥0.6 = "怎么用都免费"的爽感；窗口倒计时 = "回满血"回访钩子。
2. **付费墙时机精准** → 只在用户**主动够到高端模型**、且明显更好用时弹（不是随机骚扰）。peak-end 体验：让他先尝到 Claude 的好，再告诉他要会员。
3. **会员专属"哇时刻"** → 把最唬人的功能（[075] 并行多小队调研 / [073] 面板应用引擎）设为 Pro+ 专属，制造"为这个值得开会员"。
4. **沉没成本黏性** → 月费已付 → 多用 → streak/成就（[045]）叠加 → 迁移成本高。
5. **年付钩子** → 年付 8 折已有；可加"年付送 ¥X 钱包额度"双重锚。
6. **额度可视化制造轻焦虑** → §2.4 进度条，临近用完时是最佳转化窗口。

## 6. 风险与对策

| 风险 | 对策 |
|---|---|
| Flash 免费层被刷（脚本党） | 演示阶段前端额度易绕过；接真后端时按账号/设备/IP 限额 + 注册门槛 |
| 高端模型成本失控 | 计费 session 余额是**硬上限**（§4.0），归零即降级 Flash，绝不垫付；高端默认锁、需后台显式开通 |
| 供应商调价打穿毛利 | §4 定价自动同步 + 加价率（×1.3）是相对值，调价自动跟涨 |
| 用户嫌"按量"算不清 | 会员包月就是解药（确定性支出）；按量页给实时预估 |
| 免费太香没人付费 | 付费墙绑定**模型质量差**而非"功能阉割"——想要更聪明就得付，这是刚需付费点 |
